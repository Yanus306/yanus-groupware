# Git / GitHub 협업 컨벤션

> 이 문서는 현재 팀의 **최신 Git / GitHub 작업 규칙**입니다.  
> 기존 문서의 예시와 충돌할 경우 **이 문서를 우선**합니다.

---

## 목적

- 팀원과의 소통을 쉽게 하기 위함
- 커밋, PR, 이슈 히스토리를 추적하기 쉽게 만들기 위함
- 작은 단위의 변경을 명확하게 남기기 위함

---

## 브랜치 전략

- `main`은 배포 브랜치입니다.
- 기능 개발은 직접 `main`에서 하지 않습니다.
- 작업은 별도 브랜치에서 진행한 뒤 `develop`으로 PR을 올립니다.
- 기본 흐름은 `작업 브랜치 → develop PR → develop 머지 → develop → main PR` 입니다.
- `develop` 머지는 작업 정리 후 진행할 수 있습니다.
- `develop` 머지는 GitHub CI 확인을 기다리지 않고 바로 진행합니다.
- `main` 머지는 배포 성격이므로 저장소 관리자만 진행합니다.
- `develop -> main` PR은 이전 배포 PR 제목/본문 형식을 따른 릴리스 PR로 열어둡니다.

### 브랜치 규칙

- 브랜치명은 **소문자**로 작성합니다.
- 팀 규칙 예시: `feat/#<이슈번호>`
- 기능별로 브랜치를 분리합니다.

**예시**

```bash
git checkout develop
git checkout -b feat/#12
```

## 커밋 메시지 컨벤션

### 기본 형식

```text
[Feat] #15 - 메인 화면 UI 와이어프레임 구현

- 변경 내용 1
- 변경 내용 2
```

### 규칙

1. 커밋 유형은 영어 대문자로 작성합니다.
2. 제목과 본문은 빈 줄로 구분합니다.
3. 제목과 본문은 한글로 작성합니다.
4. 제목 첫 글자는 대문자로 작성합니다.
5. 제목 끝에는 마침표를 붙이지 않습니다.
6. 제목은 영문 기준 50자 이내로 작성합니다.
7. 본문에는 **무엇을 왜 바꿨는지**를 적습니다.
8. 여러 항목이 있으면 글머리 기호를 사용합니다.
9. 한 커밋에는 한 가지 변경만 담습니다.
10. 기능 단위보다 큰 커밋은 만들지 않습니다.
11. 작은 기능 하나가 끝날 때마다 바로 커밋합니다.
12. 여러 화면, 여러 기능, 여러 성격의 변경을 한 커밋에 묶지 않습니다.
13. 커밋 메시지와 본문에 비밀번호, 토큰, 이메일, 서버 주소의 내부 정보, 개인 식별 정보 같은 민감한 내용을 절대 적지 않습니다.
14. 실제 계정, 테스트 계정, API 응답에 포함된 민감 정보는 커밋 메시지와 PR 본문에서 모두 마스킹합니다.

### 커밋 유형

| 유형 | 의미 |
|------|------|
| `Feat` | 새로운 기능 추가 |
| `Fix` | 버그 수정 |
| `Docs` | 문서 수정 |
| `Style` | 코드 포맷팅, 세미콜론 누락 등 코드 동작 변화 없음 |
| `Refactor` | 코드 리팩토링 |
| `Test` | 테스트 코드 추가 및 수정 |
| `Chore` | 패키지 매니저 수정, 설정 변경, 기타 잡무 |
| `Comment` | 필요한 주석 추가 및 변경 |
| `Rename` | 파일 또는 폴더 이름 변경만 수행 |
| `Remove` | 파일 삭제만 수행 |
| `!HOTFIX` | 긴급 치명적 버그 수정 |
| `CICD` | CI/CD 관련 설정 |

### 좋은 예시

```text
[Feat] #12 - 회원가입 화면 유효성 검증 추가

- 이메일 형식 검증 로직 추가
- 비밀번호 확인 불일치 에러 문구 추가
- 잘못된 입력을 초기에 차단해 가입 실패를 줄이기 위해 반영
```

```text
[Test] #12 - 회원가입 폼 유효성 검사 테스트 추가

- 이메일, 비밀번호, 비밀번호 확인 케이스 검증
- 기능 구현 전 실패 테스트를 먼저 고정하기 위해 작성
```

### 피해야 하는 예시

```text
회원가입 수정
```

```text
[Feat] #12 - signup page update.
```

```text
[Feat] #12 - 회원가입, 로그인, 프로필 수정 한 번에 작업
```

### CLI에서 여러 줄 커밋 메시지 작성

```bash
git commit -m "[Feat] #3 - 회원가입 기능 추가

- 회원가입 API 연결
- 입력 검증 UI 추가"
```

## PR 컨벤션

- PR 제목은 커밋 메시지 형식을 따릅니다.
- PR 제목은 소문자 규칙을 따른 팀 표기와 충돌하지 않도록 팀 내에서 합의한 형식으로 통일합니다.
- 현재 저장소에서는 제목에 이슈 번호를 함께 적는 방식을 권장합니다.
- 작업 브랜치에서 `develop`으로 PR을 생성합니다.
- `develop` 머지는 검증 완료 후 진행할 수 있습니다.
- `develop` PR은 로컬 검증 기준으로 머지하며, GitHub CI 완료를 기다리지 않습니다.
- `main` 머지는 저장소 관리자만 진행합니다.
- GitHub UI에서는 `.github/PULL_REQUEST_TEMPLATE.md` 템플릿을 기본으로 사용합니다.

**예시**

```text
[feat] #12 - 메인 와이어 프레임 구현
```

### main 배포 PR 규칙

- `develop -> main` PR 제목은 반드시 `release: v<major>.<minor>.<patch>` 형식으로 작성합니다.
- 현재 저장소의 `v1.0.9` 다음 배포는 `v1.0.10`이 아니라 `v1.1.0`으로 올립니다.
- 즉, patch 숫자가 `10`이 되는 시점에는 patch를 `0`으로 되돌리고 minor를 1 올립니다.
- 예시: `v1.0.8 → v1.0.9 → v1.1.0 → v1.1.1`
- 본문은 이전 `develop -> main` PR 형식을 따릅니다.
- 권장 섹션은 `작업 내용`, `변경 이유`, `상세 변경 사항`, `테스트`, `참고`, `리뷰 포인트`, `관련 이슈`입니다.
- `main` PR은 열어두기만 하고, 별도 지시가 없으면 머지하지 않습니다.

### PR 본문 권장 항목

- 무엇을 변경했는지
- 왜 변경했는지
- 테스트 여부
- 리뷰 시 확인 포인트

## 이슈 컨벤션

- 기능별로 이슈를 분리해서 생성합니다.
- 한 이슈는 한 기능 또는 한 작업 단위를 담당합니다.
- 이슈 제목은 소문자로 통일합니다.
- GitHub UI에서는 `.github/ISSUE_TEMPLATE/` 아래 템플릿을 사용합니다.

**예시**

```text
[feat] 아이디 비밀번호 찾기 와이어 프레임 구현
```

---

## 작업 단위 원칙

- 한 커밋에는 한 가지 문제만 담습니다.
- 투두리스트 한 항목 단위로 커밋한다고 생각합니다.
- 추적 가능성과 리뷰 효율을 위해 큰 작업을 잘게 나눕니다.
- 기능 개발은 **테스트 작성 → 구현 → 리팩토링** 흐름을 지킵니다.
- 작은 기능 단위 커밋은 선택이 아니라 필수입니다.
- 커밋이 커졌다고 느껴지면 더 쪼갠 뒤 다시 커밋합니다.
- 가능하면 `테스트 커밋 → 구현 커밋 → 리팩토링 커밋`으로 나눕니다.
- 하나의 커밋에서 여러 기능을 동시에 끝내려는 방식은 지양합니다.
- 민감한 운영 정보, 계정 정보, 토큰 값, 개인 정보는 커밋 메시지와 본문에 남기지 않습니다.

## 코드 리뷰 컨벤션

- `P1)` 반드시 수정이 필요한 이슈
- `P2)` 참고 또는 권장 사항

**예시**

```text
P1) 이 분기에서는 null 케이스가 누락되어 런타임 오류가 발생할 수 있습니다.
```

```text
P2) 고생하셨습니다. 이 로직은 다음 단계에서 공통 훅으로 분리해도 좋겠습니다.
```

---

## 요약

- 개발은 별도 브랜치에서 진행하고 `develop`으로 PR을 올립니다.
- `main`은 배포 브랜치이며 머지는 저장소 관리자가 진행합니다.
- 커밋, PR, 이슈는 팀 컨벤션에 맞춰 통일된 형식으로 작성합니다.
- 가장 작은 기능 단위로 나누고, 테스트를 먼저 작성합니다.
