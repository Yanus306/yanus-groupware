# 테스트 트러블슈팅

> 최근 프로젝트에서 실제로 겪은 테스트 이슈와 해결 방법을 정리한 문서입니다.

## 1. 시나리오 테스트가 정렬 변경 후 깨짐

증상:
- `getAllByRole(...)[0]` 같은 인덱스 기반 선택이 실패
- 정렬 규칙 변경 후 다른 행을 클릭함

해결:
- 행 이름이나 셀 텍스트로 대상 행을 먼저 찾고 `within()`으로 버튼을 찾습니다.

권장 예시:

```tsx
const memberRow = screen.getByRole('row', { name: /김멤버/ })
await user.click(within(memberRow).getByRole('button', { name: /팀 변경/ }))
```

## 2. 성공 메시지 텍스트가 바뀌어 테스트 실패

증상:
- 대상 사용자 이름이 바뀌었는데 예전 고정 문자열을 기대함

해결:
- 실제로 조작한 대상 이름을 기준으로 기대 문자열을 맞춥니다.
- 가능하면 “대상 행 선택 → 액션 → 결과 메시지” 흐름을 같은 데이터 기준으로 묶습니다.

## 3. Playwright가 팀 변경 모달 옵션을 못 찾음

증상:
- 모달은 떠 있는데 선택자 매칭 실패

원인:
- 페이지마다 팀 선택 모달 클래스가 다를 수 있음
- 예전 클래스명에 테스트가 묶여 있었음

해결:
- 모달을 먼저 좁히고 그 안에서 옵션을 찾습니다.

## 4. Playwright 전체 실행 시 로그인 실패

증상:
- 개별 실행은 되는데 전체 실행에서는 간헐적으로 `서버 오류가 발생했습니다`

원인:
- 실서버 공용 계정을 여러 워커가 동시에 사용
- 상태 변경과 토큰 발급이 겹치며 불안정해짐

해결:
- `playwright.config.ts`에서 `workers: 1`로 직렬 실행

## 5. localStorage와 서버 값이 충돌

증상:
- 근무 일정이 저장된 뒤에도 예전 localStorage 값처럼 보임

해결:
- 서버 응답을 우선값으로 사용
- localStorage는 네트워크 실패 시 fallback 으로만 사용

## 6. 권한 테스트는 통과하는데 실서버 API는 허용됨

증상:
- 프론트 버튼은 숨겨져 있는데 직접 API 호출은 성공

해결:
- 프론트 테스트와 별개로 백엔드 권한 검증이 필요
- 이 항목은 [backend-security-checklist](../guides/backend-security-checklist.md)에서 따로 관리

## 7. 테스트 간 상태 오염

증상:
- 앞 테스트의 localStorage, mock 데이터, 로그인 상태가 다음 테스트에 남음

해결:
- `beforeEach`에서 데이터 초기화
- `afterEach`에서 `localStorage.clear()`
- MSW 핸들러와 가변 배열 상태를 매 테스트마다 복구

## 8. 어떤 테스트를 먼저 써야 할지 애매함

기준:
- 계산 규칙이 핵심이면 단위 테스트
- 화면 행동이 핵심이면 RTL 통합 테스트
- 여러 페이지를 잇는 흐름이면 앱 시나리오 테스트
- 실서버 최종 확인이면 Playwright E2E
