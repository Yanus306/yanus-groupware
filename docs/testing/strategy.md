# 테스트 전략

> 현재 프로젝트에서 사용하는 최신 테스트 전략을 정리한 문서입니다.

## 목표

- 작은 기능 단위로 빠르게 실패를 감지합니다.
- 로그인 이후 공통 프리로드와 권한 흐름을 안정적으로 검증합니다.
- mock 기반 테스트와 실서버 E2E를 분리해 각각의 역할을 명확히 합니다.

## 기본 원칙

- 개발은 항상 `RED → GREEN → REFACTOR` 순서로 진행합니다.
- 커밋은 `테스트 → 구현 → 리팩토링`을 가장 작은 단위로 나눕니다.
- 한 테스트는 한 사용자 가치 또는 한 규칙만 검증합니다.
- 공통 로직은 단위 테스트, 화면 흐름은 통합 테스트, 배포 전 핵심 시나리오는 E2E로 검증합니다.

## 테스트 레이어

### 1. 단위 테스트

대상:
- 유틸 함수
- 권한 계산 함수
- API 응답 매핑 로직
- 상태 가공 함수

도구:
- Vitest

기준:
- 입력과 출력이 분명한 순수 로직은 단위 테스트 우선
- 정렬, 권한, 포맷팅, 필터링은 단위 테스트로 먼저 고정

### 2. 컴포넌트/통합 테스트

대상:
- 페이지와 provider 조합
- 로그인 후 데이터 로드
- 멤버 관리 액션
- 대시보드 상호작용

도구:
- React Testing Library
- MSW

기준:
- 사용자 행동 중심으로 검증
- 실제 텍스트, 버튼, 라우트 변화를 확인
- 서버는 MSW로 모사하되 성공/실패/권한 케이스를 모두 포함

### 3. 앱 시나리오 테스트

대상:
- 로그인
- 대시보드 진입
- 멤버 관리
- 팀 변경
- 로그아웃

기준:
- 여러 페이지를 가로지르는 대표 흐름을 하나의 테스트로 연결
- 정렬, 권한, 성공 메시지처럼 실제 화면 규칙이 바뀌면 시나리오 테스트도 같이 갱신

### 4. Playwright E2E

대상:
- 로그인과 대시보드
- 관리자 멤버 팀 변경 후 원복
- 관리자/팀장/멤버 권한별 접근 제어

기준:
- 실서버 기준 검증
- 공용 계정 사용으로 인해 직렬 실행 유지
- 환경 변수로만 계정 정보를 주입

실행:

```bash
npm run test:e2e
```

## 로그인 후 데이터 로드 전략

- 로그인 직후 공통 프리로드에서 사용자 기준 핵심 데이터를 먼저 로드합니다.
- 각 페이지는 추가 데이터만 보강 fetch 합니다.
- 일정/할 일처럼 홈에서 바로 보이는 데이터는 첫 진입 시 준비되어 있어야 합니다.
- 서버 데이터가 기준이며, localStorage는 오프라인/실패 fallback 용도로만 사용합니다.

## 권한 테스트 전략

- 권한 계산은 `shared/lib/permissions.ts` 단위 테스트로 고정합니다.
- 관리자/팀장/멤버 화면 노출은 컴포넌트 테스트로 확인합니다.
- 라우트 접근 차단은 Playwright E2E로 한 번 더 검증합니다.

## 테스트 작성 체크리스트

- 실패하는 테스트를 먼저 만들었는가
- 테스트 대상이 하나의 기능 단위로 충분히 작은가
- 성공 케이스와 실패 케이스가 모두 있는가
- 권한/로딩/빈 상태가 필요한 화면인가
- 메시지나 정렬 규칙이 바뀌면 관련 테스트도 같이 고쳤는가

## 커밋 전략과 테스트 연결

- `test`: 실패 테스트 추가
- `feat` 또는 `fix`: 테스트를 통과시키는 최소 구현
- `refactor`: 구조 정리 후 회귀 확인

이 순서를 커밋 단위로 유지하는 것을 기본 원칙으로 합니다.
